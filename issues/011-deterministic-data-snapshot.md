# 011：确定性数据快照与最终结论一致性

## 状态

v0.2.29 代码已实现，自动化回归与 Web 健康检查通过；真实 A 股全链路验收尚未通过，不应部署。

## 问题

旧架构由七位分析师模型自行决定是否调用数据工具。模型能力、提示词理解或并行消息状态都可能造成“本应采集但完全未调用”，门控只能事后发现，无法保证分析开始前已经取得完整数据基础。最终组合经理的自由文本还可能出现“抬头评级为持有、下方操作指令为卖出”等内部矛盾。

## 方案

1. `data_snapshot.py` 根据分析范围生成固定请求计划，在任何模型运行前逐项执行。
2. 七项分析固定覆盖 17 类数据、24 个请求，其中技术指标为 8 个独立请求；三项速览覆盖 10 类数据、17 个请求。
3. 每条记录保存请求编号、中文栏目名、适用分析师、状态、采集时间、耗时和可用正文；失败正文清空，底层错误、代理、网址及参数不进入产品快照。
4. 同一份共享数据只采一次，随后按分析师过滤成只读子集。分析师不绑定工具，只能解释快照；失败栏目必须写明无法判断，正常无记录不算缺失。
5. 质量门控继续基于逐请求台账计算数据可信度和结论边界；快照与台账一并持久化。
6. 最终文本经过代码校验：只允许唯一的 Buy / Overweight / Hold / Underweight / Sell 评级，并核对明确操作方向。低数据质量或文本冲突时，网页、历史和导出不展示操作结论。
7. 新图删除了旧工具节点，因此断点线程命名空间升级；升级前的未完成任务从头运行，避免恢复到不存在的节点。

## 图结构

`START → Data Snapshot → 并行 Analyst → Quality Gate → 多空辩论 → Trader → 风险辩论 → Portfolio Manager → 代码结论校验`

## 验收标准

- 固定计划与启用分析师的应采工具集合完全一致，新增应采工具但漏改计划时测试或运行立即失败。
- 每个计划请求恰好执行一次；失败和正常无记录语义不混淆。
- 分析师只看到自己的快照栏目，且没有工具绑定路径。
- 报告持久化完整快照、数据台账、数据质量和校验后结论。
- “Rating: Hold + 决策：卖出”等矛盾被代码拦截。
- 全量自动化测试、Web 启动检查及至少一只真实 A 股全链路案例通过后，方可标记完成并部署。
