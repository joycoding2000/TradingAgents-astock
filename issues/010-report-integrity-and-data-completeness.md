# 010：分析报告一致性与关键数据完整性

## 背景

2026-07-18 在阿里云真实运行 `600879`、`300219`、`300184` 后审查完整结果。发现系统虽然已记录数据台账，但关键数据失败时仍可把模型生成的买卖结论、价位和投入比例展示给用户；历史页还可能从较早的研究建议中读出与最终风控决定不同的信号。

这不符合“数据是结论基石”的产品原则。关键数据可用应是设计目标，数据缺失只能是受控降级状态，不能形成交易引导。

## 真实报告发现

| 标的 | 数据状态 | 发现 |
|---|---|---|
| 600879 | 低 | 行业对比、个股资金流失败；页面仍可展示最终卖出和具体操作。 |
| 300219 | 高 | 数据台账完整，但周末 `2026-07-18` 被当作交易日，报告出现“今天/明天开盘”的误导性时点表述。 |
| 300184 | 低 | 个股资金流、行业对比失败；仍出现卖出指令和具体价位。 |

另有横向问题：旧历史页按 `investment_plan → trader → final` 的顺序查找 `BUY/SELL/HOLD`，会把前面 Hold 当作标题，即使最终结论为 Sell。

## v0.2.25 处理

1. 关键台账失败后，Web、Markdown、PDF 的唯一用户结论为“数据不完整”；隐藏所有可能包含交易动作的分析、辩论和最终决策正文。原始状态 JSON 保留作审计。
2. 低可信度运行不把模型原始买卖意见写入经验记忆，防止下一次分析把不完整结论当先验。
3. 历史页只解析 `final_trade_decision`，支持 Buy / Overweight / Hold / Underweight / Sell 五档；低可信度强制显示数据不完整。
4. 东财请求使用全周期串行锁；429、5xx、资金流历史列表为空和行业列表为空均按失败处理，由台账和门控接收。
5. 动态代理切换前关闭旧 Session 连接池；单个请求最多轮换一次，避免付费代理异常时无限扣费。

## 后续验收标准

- 以正常交易日运行 `600879`、`300219`、`300184`：所有关键工具的最终状态均为“成功”或业务定义的“正常空结果”，不得有“失败”。
- 若任一关键工具失败，所有用户出口（实时页面、历史页、Markdown、PDF）不得出现买入/卖出、目标价、止损价、投入比例等操作内容。
- 周末和节假日的报告须使用“最近交易日/下一个交易日”措辞；不能把非交易日表述为“今天开盘”。
- 单项分析只能陈述该角度证据和待核实项，不得把新闻缺失、接口失败或正常空结果推导为公司/股东的负面事实。
